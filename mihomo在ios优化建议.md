# iOS 端 libmihomo 稳定性与轻量化优化建议 (致 Go 核心研发团队)

**背景说明：**
当前 iOS 客户端在通过 NetworkExtension (PacketTunnelProvider) 接入 `libmihomo.a` 时，遇到了严重的稳定性问题。高频并发下频繁触发底层 `EXC_BAD_ACCESS` 崩溃（多表现为 `mach_msg_trap` / `CppObjectLocks` 报错），并伴随 iOS Jetsam 内存溢出强杀。

为了实现 iOS 端（特别是 NetworkExtension 进程下）的稳定运行，需要对 CGO 交互层的内存管理进行重构，并对内核进行 iOS 专属的轻量化裁剪。以下是具体的优化点和需要避免的坑。

---

## 核心优化点 1：重构 CGO 跨语言内存生命周期 (解决致命崩溃)1

**当前痛点（大坑）：**
目前 iOS 扩展端每次收到网络包，都调用 `MobileNewPacketFlowPacket` 创建一个由 gomobile 包装的对象，再通过 `MobileFeedPacketFromFlow` 喂给内核。
* **致命冲突 (Double Free / 悬垂指针)**：`MobileNewPacketFlowPacket` 返回的对象生命周期被 iOS 端的 Swift ARC (自动引用计数) 接管。当 Go 内核处理完数据包并在底层释放内存后，Swift 端的 ARC 往往会存在延迟释放。当 ARC 最终触发清理时，会调用 C++ 析构函数，但此时底层的 Go 指针已经失效，直接导致内核互斥锁 (`CppObjectLocks`) 崩溃。
* **性能瓶颈**：针对每一个 MTU (约 1500 字节) 的高频网络包，都在跨语言边界疯狂创建和销毁对象，这在 iOS 严格的内存限制下是不可接受的。

**优化建议：**
**废弃高频数据流的对象传递模式，改用纯字节数组 (Raw Bytes) 交互。**

请提供以下新的 API 供 iOS 端调用：
```go
// 建议新增的 API 设计：
// 直接传递 []byte 字节流和协议族 (AF)
// 内核侧内部自行拷贝或消费，不再返回对象给 Swift，彻底切断两端生命周期绑定
func MobileFeedPacketBytes(data []byte, af int64) bool 

// （可选：如果追求极致性能，可提供批量喂流接口，减少 CGO 调用次数）
func MobileFeedPacketsBytesBatch(data [][]byte, afs []int64) int 
```

## 核心优化点 2：内存占用与 GC 优化 (避免 Jetsam 15MB 强杀)

**当前痛点（大坑）：**
iOS 的 NetworkExtension (VPN 进程) 极其脆弱，系统最高仅允许 **15MB** 左右的内存占用。一旦超过该阈值或内存暴涨过快，进程会被系统无情秒杀（Jetsam Kill），表现为 VPN 频繁断开重连。

**优化建议：**
1. **禁用大内存预分配**：检查内核初始化时，绝对不能有预分配过大 Buffer 的行为。
2. **强制复用内存池 (sync.Pool)**：针对高频数据包处理（Packet 读取、加解密等），必须全局强制使用 `sync.Pool` 进行 `[]byte` 缓冲区的复用。如果每次都 `make([]byte)`，会产生海量短生命周期对象，导致 Go GC 频繁触发产生尖刺，直接触发 Jetsam Kill。
3. **主动释放闲置内存**：建议在内核内部，当检测到网络空闲或通过定时器（如每 1-2 分钟），主动调用一次 `debug.FreeOSMemory()`，强制将闲置内存归还给 iOS 操作系统。

## 核心优化点 3：核心功能裁剪与体积优化 (按需编译)

**当前痛点（大坑）：**
完整版内核包含大量对 iOS VPN 扩展无用的模块，这些模块会增加二进制体积、延长启动时间、并无端占用宝贵的内存资源。

**优化建议：**
请通过 Go 的 `build tags` 或代码裁剪，移除 iOS 扩展端不必要的模块：
1. **避免加载庞大的静态资源**：iOS 客户端目前已**移除**本地读取 `Country.mmdb` 的依赖。请确保内核不要在启动时强制将巨大的 MMDB 文件全量加载进内存（建议按需查询或精简数据库）。
2. **移除 UI 控制台/外部 API**：扩展内部不需要运行 HTTP 外部控制 API 服务器 (RESTful API)，请裁剪这部分代码。
3. **编译参数优化**：在构建 `libmihomo.a` 时，请务必开启体积优化参数：
   * 使用 `-ldflags="-s -w"` 剔除调试符号表。
   * 配合使用 `gomobile bind -trimpath` 减小最终包体积。

## 核心优化点 4：并发与连接数控制 (避免句柄耗尽)

**当前痛点（大坑）：**
iOS 系统对单个进程（特别是后台扩展进程）打开的文件描述符（FD）和并发连接数有严格的硬性限制。如果内核像桌面端一样无节制地处理多路复用或保留过多空闲连接，极易导致 `too many open files` 错误，甚至直接被系统强制挂起。

**优化建议：**
1. **控制并发连接上限**：在内核级别增加硬性限制，限制最大并发 TCP/UDP 连接数（建议 iOS 端限制在 1000 以内）。
2. **激进的连接超时策略**：大幅缩短空闲连接（Idle Connection）的保活时间。桌面端可以保持几分钟甚至更久，但在 iOS 上，建议空闲超过 15-30 秒即可强制断开，及时回收 FD 句柄。
3. **优化 DNS 查询并发**：避免一次性发起海量并发 DNS 查询，这会导致瞬间创建大量 UDP 句柄，触发系统风控。

## 核心优化点 5：日志级别与 IO 损耗控制

**当前痛点（大坑）：**
在 iOS 扩展中高频写入日志（不管是写入本地文件还是通过 CGO 回传给 Swift 打印）会产生极大的 IO 开销和 CPU 损耗，引发发热和耗电问题，最终可能触发 Watchdog 超时崩溃。

**优化建议：**
1. **默认强制静默模式**：Release 包（交付给 iOS 的 `.a`）必须默认设置日志级别为 `error` 或 `silent`。
2. **切断高频 CGO 日志回调**：如果当前存在内核向 Swift 层实时推送日志的 CGO 回调，请务必将其彻底关闭或设为按需开启。千万不要将每一个数据包的流转信息打印出来。

## 核心优化点 6：iOS 休眠唤醒与网络切换处理

**当前痛点（大坑）：**
iOS 设备在锁屏休眠（Sleep）或在 Wi-Fi/5G 之间切换时，底层网络接口会发生断崖式变化。如果 mihomo 内核没有感知或处理不当，会导致旧的 Socket 连接全部死锁（Deadlock），唤醒后 VPN 无法恢复流量，只能重启。

**优化建议：**
1. **网络变更重置机制**：内核需要提供一个轻量级的 `MobileResetNetwork()` 或 `MobileRestartTunnel()` 接口。当 iOS 侧监听到网络切换或系统唤醒时，调用此接口，让内核快速清空已失效的 TCP/UDP 连接池和 DNS 缓存，而无需销毁整个 mihomo 实例。
2. **Socket 异常检测 (Keep-Alive)**：在底层 Dialer 必须开启 TCP Keep-Alive，并设置较短的探测时间，以便在网络环境断开时迅速销毁无效的 FD，防止协程堆积卡死。

---

## 核心优化点 7：DNS 污染与系统 DNS 接管防死循环

**当前痛点（大坑）：**
在 iOS PacketTunnel 中，如果 mihomo 内部发出的 DNS 请求没有被正确标记（或绕过 TUN 路由），会被重新吸入 TUN 虚拟网卡，导致 DNS 流量死循环，瞬间打满 CPU 并耗尽内存。

**优化建议：**
1. **必须使用系统级 BindToDevice 或 SO_MARK**：内核层发出的真实物理网络请求（包括 DNS UDP 查询和出站 TCP 连接），必须确保能够绑定到主网卡（如 `en0` / `pdp_ip0`），绕过 TUN 接口。
2. **Fallback DNS 策略**：当检测到并发 DNS 查询超时激增时，需有防雪崩机制，直接丢弃过多的查询，防止上层业务无限制重试压垮内核。

## 核心优化点 8：UDP 穿透与 QUIC 协议在 iOS 上的特殊处理

**当前痛点（大坑）：**
iOS 系统的蜂窝网络（特别是国内运营商的 5G/4G）对 UDP 流量有严格的 QoS 限制和极短的 NAT 保活时间。如果内核完全依赖 UDP 承载底层代理（例如单纯使用 QUIC 或无伪装的 UDP 节点），在锁屏或后台极易被运营商阻断，导致假死。同时，高频 UDP 转发在 iOS 虚拟网卡中比 TCP 消耗更多 CPU。

**优化建议：**
1. **UDP 智能回退机制 (Fallback to TCP)**：如果节点支持 UDP/QUIC，内核必须具备快速感知 UDP 阻断并回退到 TCP 承载的能力，而不是一直死等 UDP 超时。
2. **TUN 层的 UDP 会话管理优化**：针对用户打游戏、语音等真实 UDP 流量，优化 TUN 层的 UDP 会话过期时间（Session Timeout）。iOS 下不宜保留过长的无活动 UDP 会话，建议空闲 30-60 秒即刻清理。

---

## 核心优化点 9：配置解析的轻量化 (避免启动 OOM)

**当前痛点（大坑）：**
目前 iOS 启动内核时，需要解析 `config.yaml`。如果这个 YAML 文件包含了数万条规则（Rules）和庞大的节点列表（Proxies），Go 内核在解析这些文本并构建内存对象树的瞬间，内存会暴涨，极易在启动阶段直接触发 OOM（内存溢出强杀）。

**优化建议：**
1. **Rule-Provider / Proxy-Provider 懒加载**：强烈建议开启并优化 Provider 机制。不要将所有规则写死在单个配置文件中，而是将外部规则列表设计为**按需加载 (Lazy Load)** 或流式读取，避免一次性分配庞大的内存树。
2. **精简内置预设**：iOS 专用的内核包应精简内置的 DNS 规则、回退组等，把解析负担降到最低。

---

## 总结：iOS 联调验收标准

1. **API 确认**：新版 `libmihomo.a` / `Mihomo.xcframework` 必须包含 `MobileFeedPacketBytes` (或类似命名的纯字节流注入方法)，不再依赖 `MobileNewPacketFlowPacket` 对象管理。
2. **内存验证**：在高负载测速（如 Youtube 4K 跑满带宽）场景下，iOS 扩展进程内存需稳定在 15MB 以下，不再触发 Jetsam 强杀。
3. **稳定性验证**：彻底解决由 `CppObjectLocks` 和 ARC 延迟释放引起的 `EXC_BAD_ACCESS` 崩溃。
